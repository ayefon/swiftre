Deployment

Run SwiftRE as a Docker container or build from source.

Docker Image

Build the image from the project root:

docker build -t swiftre .

Run it:

docker run -p 8080:8080 swiftre

The API is now available at http://localhost:8080.

Docker Compose

For persistent storage and easier management, use Docker Compose:

docker compose up -d

The docker-compose.yml:

services:
  swiftre:
    build: .
    image: swiftre
    ports:
      - "8080:8080"
    volumes:
      - swiftre-data:/app/data
    environment:
      - SWIFTRE_DATA_DIR=/app/data/rulesets

volumes:
  swiftre-data:

Persistent Storage

The swiftre-data volume persists rulesets across container restarts. Without a volume, rulesets are stored in memory and lost when the container stops.

Environment Variables

VariableDefaultDescription
SWIFTRE_DATA_DIR(none)Directory for persistent ruleset storage. If unset, rulesets are in-memory only.

Building from Source

Requires Swift 6.0+ toolchain:

swift build -c release
swift run SwiftREServer serve

The server binds to 0.0.0.0:8080 by default. Override with Vapor's command-line options:

swift run SwiftREServer serve --hostname 127.0.0.1 --port 9090

Accessing the Server

Production Considerations

Reverse Proxy

In production, place a reverse proxy (nginx, Caddy, etc.) in front of SwiftRE for TLS termination, rate limiting, and access control. SwiftRE itself does not provide authentication.

CORS

If your frontend is served from a different origin than the API, you'll need to configure CORS headers. This can be done at the reverse proxy level.

Persistent Rulesets

Always use a Docker volume or the SWIFTRE_DATA_DIR environment variable in production. Without persistent storage, all rulesets are lost on container restart.

No authentication: SwiftRE does not include authentication or authorization. Anyone who can reach the API can create, modify, and delete rulesets. Protect the API with a reverse proxy or network-level access controls in production.
