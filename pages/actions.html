Actions

Actions define what happens when a rule fires. Multiple actions in a single rule are separated by AND.

Message Actions

These actions emit events with a message string. The event kind determines how the message should be treated by the consuming application.

ERROR

Emit an error event. Typically used for validation failures that must be corrected.

THEN ERROR "Email is required"

WARNING

Emit a warning event. Useful for non-blocking issues the user should be aware of.

THEN WARNING "This order exceeds the typical limit"

INFO

Emit an informational event. Used for feedback, confirmations, or helpful context.

THEN INFO "Welcome, {firstName}!"

Field-targeted Variants (ON)

Associate a message action with a specific field using the ON keyword. This enables precise field highlighting in form UIs without relying on message text matching:

THEN ERROR ON email "A valid email is required"
THEN WARNING ON quantity "Quantity seems unusually high"
THEN INFO ON status "This field was auto-populated"

When a field-targeted event is returned, the response includes a field property:

{
  "kind": "error",
  "ruleName": "Email Required",
  "field": "email",
  "message": "A valid email is required"
}

Message Interpolation

Use {factPath} inside message strings to insert the current value of a fact:

THEN INFO "Welcome, {firstName}! Your order total is {orderTotal}."
THEN ERROR ON email "The address {email} appears to be invalid"

Interpolation works with dotted paths: {address.city}.

Data Modification: SET

Modify a fact value in working memory. In iterative mode, this can trigger re-evaluation of rules that depend on the modified field.

Setting Literal Values

THEN SET status TO "approved"
THEN SET retryCount TO 0
THEN SET isVerified TO true

Date Arithmetic in SET

THEN SET expiryDate TO TODAY + 30 DAYS
THEN SET renewalDate TO `contractStart` + 1 YEAR
THEN SET reminderDate TO `deadline` - 7 DAYS

Date Difference in SET

Compute the integer span between two dates and store the result:

THEN SET duration TO DAYS BETWEEN startDate AND endDate
THEN SET tenure TO YEARS BETWEEN hireDate AND TODAY
THEN SET monthsRemaining TO MONTHS BETWEEN TODAY AND expiryDate

Signal Events: SIGNAL

Emit a custom event for application-level event handlers. Signal events are not displayed in form UIs — they are passed through in the response for downstream processing.

THEN SIGNAL AN EVENT OF TYPE approvalRequired
  WITH A PAYLOAD OF "Order {orderId} needs manager approval"

The type can be a bare word or a quoted string:

THEN SIGNAL AN EVENT OF TYPE "audit-log"
  WITH A PAYLOAD OF "User {userId} changed status to {status}"

In the response, signal events appear with kind: "signal" and the custom type in the payload:

{
  "kind": "signal",
  "ruleName": "Approval Required",
  "message": "Order ORD-123 needs manager approval",
  "payload": {
    "type": "approvalRequired"
  }
}

Design intent: Signal events provide inherent extensibility. Your application registers event handlers for recognized types and ignores or passes through unrecognized ones. No engine modifications are needed to add new application behaviors.

Example Use Cases

  • Triggering workflow steps (e.g., sending an order for approval)
  • Audit logging (e.g., recording who changed what and when)
  • Sending notifications (e.g., email or Slack alerts)
  • Integrating with external systems (e.g., CRM updates)

Combining Actions

Chain multiple actions with AND:

RULE "High-Value Order" PRIORITY 5
WHEN orderTotal IS GREATER THAN 10000
THEN WARNING ON orderTotal "High-value order requires review"
  AND SET requiresApproval TO true
  AND SIGNAL AN EVENT OF TYPE approvalRequired
    WITH A PAYLOAD OF "Order {orderId} total: {orderTotal}"
