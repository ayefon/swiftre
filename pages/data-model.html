Data Model

SwiftRE evaluates rules against a working memory — a flat key-value store populated from the JSON facts you submit.

Working Memory

When you execute a ruleset, you provide a JSON object of facts. SwiftRE loads these into working memory, which is a flat dictionary of fact paths to typed values. Rules read from and (via SET actions) write to working memory.

// Input JSON
{
  "firstName": "Jane",
  "age": 28,
  "email": "jane@example.com"
}

// Working memory (flat key-value pairs)
firstName → "Jane"    (string)
age       → 28        (integer)
email     → "jane@example.com" (string)

JSON Flattening

Nested JSON objects are flattened to dot-notation paths. This allows rules to reference deeply nested data using dotted field names:

// Input JSON
{
  "customer": {
    "name": "Jane",
    "address": {
      "city": "Springfield",
      "zip": "62704"
    }
  }
}

// Working memory
customer.name         → "Jane"
customer.address.city → "Springfield"
customer.address.zip  → "62704"

In rules, use backtick-quoted field references for dotted paths:

WHEN `customer.address.city` IS "Springfield"
THEN INFO "Local customer detected"

Type System

Each fact value has one of the following types:

TypeJSON SourceExample
stringJSON string"hello"
integerJSON number (no decimal)42
doubleJSON number (with decimal)3.14
booleanJSON booleantrue, false
dateISO 8601 string (coerced)"2025-06-15"
nullJSON null or missing keynull

Type Coercion

  • Numeric: Integers and doubles are compared numerically across types. 42 (integer) equals 42.0 (double).
  • String-to-date: When a string value is compared with a date operator (IS BEFORE, IS AFTER, etc.) or a date literal, SwiftRE automatically attempts ISO 8601 parsing.

Synonyms

Synonyms map human-friendly names to canonical fact paths. This lets you write rules with readable field names even when the underlying data uses technical paths:

// Synonym mapping
"customer age"  → "customer.demographics.age"
"zip code"      → "customer.address.zip"

With synonyms configured, rules can use either form:

WHEN `customer age` IS LESS THAN 18
THEN ERROR "Must be 18 or older"

Synonyms are managed per-ruleset via the API:

PUT /api/v1/rulesets/my-rules/synonyms
{
  "synonyms": {
    "customer age": "customer.demographics.age",
    "zip code": "customer.address.zip"
  }
}

Facts in the Response

The execution response includes a facts object containing the complete working memory state after execution. If SET actions modified values, the updated values appear here:

{
  "events": [...],
  "facts": {
    "status": "approved",
    "reviewDate": "2025-07-15",
    "originalStatus": "pending"
  }
}

Schema Validation

You can optionally declare a data model schema for a ruleset. The schema defines expected field types and whether fields are required. Schema validation produces warnings (not errors) when rule text references unknown fields or uses fields inconsistently:

PUT /api/v1/rulesets/my-rules/schema
{
  "fields": {
    "age":   {"type": "integer", "required": true, "description": "Customer age"},
    "email": {"type": "string",  "required": true, "description": "Contact email"},
    "score": {"type": "double",  "required": false, "description": "Optional risk score"}
  }
}

When validating rule text against a ruleset with a schema, the response may include schemaWarnings for fields not declared in the schema.
