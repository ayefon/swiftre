Execution Modes

SwiftRE supports two execution modes that control how rules are evaluated: single-pass and iterative.

Single-Pass Mode

The default mode. All rules are evaluated once, in priority order. No re-evaluation occurs, even if SET actions modify facts that other rules depend on.

POST /api/v1/rulesets/my-rules/execute
{
  "facts": {"age": 16, "email": null},
  "mode": "singlePass"
}

When to use: Form validation, simple checks, read-only rules (no SET actions), or when you want predictable single-cycle behavior.

Iterative Mode

Rules are evaluated repeatedly until no more changes occur (quiescence). Each time a SET action modifies a fact, the engine re-evaluates rules that depend on the changed field.

POST /api/v1/rulesets/my-rules/execute
{
  "facts": {"orderTotal": 150, "customerType": "gold"},
  "mode": {"iterative": 10}
}

The number (10 in this example) is the maximum number of iterations allowed.

How Iterative Mode Works

  1. Initial pass

    All rules are evaluated in priority order. SET actions modify working memory and record which fact paths changed.

  2. Selective re-evaluation

    Only rules whose conditions depend on changed fact paths are re-evaluated. This is more efficient than re-evaluating all rules.

  3. Repeat until stable

    If re-evaluation produces more changes, the cycle continues. Evaluation stops when one of three conditions is met.

Termination Conditions

ReasonDescription
completeQuiescence. No SET actions fired during the last pass, or no rules depend on the changed facts. The system has reached a stable state.
maxIterationsReachedThe iteration count hit the configured maximum. The current state is returned, but it may not be fully stable.
cycleDetectedThe engine detected that the working memory returned to a previously seen state, indicating an infinite loop. Execution stops to prevent runaway evaluation.

Example: Chained SET Rules

RULE "Apply Discount" PRIORITY 10
WHEN `customerType` IS "gold" AND orderTotal IS GREATER THAN 100
THEN SET discount TO 15

RULE "Calculate Final" PRIORITY 5
WHEN discount IS SET AND discount IS GREATER THAN 0
THEN SET finalTotal TO orderTotal
  AND INFO "Discount of {discount}% applied"

In single-pass mode, if "Apply Discount" fires first (higher priority) and sets discount to 15, "Calculate Final" will also evaluate in the same pass because it comes later in priority order.

In iterative mode, the engine recognizes that discount changed and re-evaluates rules that depend on it, ensuring "Calculate Final" fires even if priority ordering would have placed it first.

Cycle Detection

The engine hashes the complete working memory state after each iteration. If a previously seen hash reappears, a cycle is detected:

RULE "Toggle A"
WHEN flag IS true
THEN SET flag TO false

RULE "Toggle B"
WHEN flag IS false
THEN SET flag TO true

This ruleset would cycle forever. The engine detects the repeating state and stops with terminationReason: "cycleDetected".

Choosing a Mode

ScenarioRecommended Mode
Form validation (ERROR, WARNING, INFO only)Single-pass
Rules with no SET actionsSingle-pass
Rules that SET values other rules depend onIterative
Multi-step calculations or derived fieldsIterative
Workflow state machinesIterative (with reasonable max)

Tip: Start with single-pass mode. Only switch to iterative if your rules use SET actions and downstream rules depend on those modified values. Set the max iterations to a reasonable number (5–20) for safety.

Response Fields

Both modes return the same response structure. The iterations and terminationReason fields tell you how execution progressed:

{
  "events": [...],
  "facts": {...},
  "iterations": 3,
  "terminationReason": "complete",
  "summary": {"errors": 0, "warnings": 1, "infos": 1, "modifications": 2, "total": 4}
}
